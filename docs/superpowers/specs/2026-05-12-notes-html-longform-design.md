# Дизайн: аннотированный longform HTML для `/notes`

Дата: 2026-05-12
Статус: design review
Проект: `/Users/sereja/Projects/products/notes-skill`

## Контекст

Сейчас `/notes` собирает подробный Markdown и HTML через `assemble.sh`, `template.html` и `dark-theme.css`. HTML технически рабочий, но ощущается как обычная простыня текста: его можно читать, но визуальная иерархия слабо помогает удерживать мысль, возвращаться к важному и быстро понять структуру материала.

Проверенные варианты с учебными карточками не подходят: они ухудшают чтение, потому что дробят цельную мысль на плашки. Нужен не “карточный конспект”, а более сильный режим longform-чтения.

## Цель

Сделать HTML-конспект похожим на хорошую аннотированную статью:

- основной текст остается цельным и читается подряд;
- визуальные элементы помогают держать нить, а не режут материал на карточки;
- таймкоды, цитаты, кейсы и важные рамки становятся заметными;
- appendix и план действий выглядят как финальные справочные разделы, а не как продолжение одной простыни.

## Не цель

- Не превращать HTML в отдельное reader-app приложение.
- Не добавлять сложное состояние, фильтры, режимы чтения или `localStorage`.
- Не менять смысл Markdown-контракта и extraction prompts ради дизайна.
- Не делать декоративный интерфейс, где визуал конкурирует с текстом.
- Не ломать печать, offline-HTML и обычное открытие файла в браузере.

## Выбранное направление

Направление: **L1, аннотированный longform**.

Техническая глубина: **I2, семантический минимум**.

Версионирование: **новый HTML должен быть отдельной темой/версией, а текущий внешний вид должен остаться доступным как `classic` для быстрого отката**.

Это значит:

- Markdown остается главным артефактом и должен быть читаемым сам по себе.
- HTML получает больше структуры на этапе сборки или постобработки.
- CSS использует эту структуру для аккуратного longform-оформления.
- JavaScript допустим только для мелких улучшений навигации, которые не нужны для понимания страницы.

## Поведение страницы

Верх страницы:

- спокойный title block;
- краткая metadata-строка, если доступна: формат, длительность, источник;
- мягкое содержание с таймкодами;
- “главная рамка” как intro-блок, если она есть в `header.md`.

Основной текст:

- каждый `##`-блок конспекта воспринимается как глава;
- таймкод в заголовке становится кликабельным и визуально отделенным;
- абзацы остаются обычными, без карточного дробления;
- `### Тезисы` не превращаются в сетку карточек, а оформляются как компактный numbered outline;
- `### Кейс` оформляется как case callout внутри главы;
- blockquote оформляется как заметная цитата;
- короткие фразы типа “Вывод”, “Запомнить”, “Главная рамка” могут стать полевой заметкой, но только если это не ломает поток чтения.

Финальные разделы:

- appendix отделяется как справочник;
- таблицы людей, ресурсов и моделей получают более спокойный табличный стиль;
- “План действий” выглядит как actionable appendix, а не как новая глава материала.

## Семантические классы

Минимальный набор классов:

- `.note-page` — общий контейнер страницы;
- `.note-hero` — верхний блок;
- `.note-toc` — содержание;
- `.note-chapter` — один смысловой блок `##`;
- `.note-time` — таймкод или диапазон;
- `.note-outline` — список тезисов;
- `.note-case` — блок кейса;
- `.note-quote` — цитата;
- `.note-margin` — короткая полевая заметка;
- `.note-appendix` — финальные справочные разделы;
- `.note-action-plan` — план действий.

Эти классы должны добавляться детерминированно. Если распознавание сомнительное, лучше оставить обычный HTML, чем угадать неверную структуру.

## Архитектура изменений

Предпочтительный путь:

1. Оставить `assemble.sh` ответственным за Markdown assembly и вызов Pandoc.
2. Сохранить текущую тему как `classic`.
3. Добавить новую тему `longform` отдельными файлами, чтобы можно было сравнить и откатиться без восстановления из git.
4. Добавить легкую HTML-постобработку после Pandoc или рядом с `assemble.sh`.
5. Постобработка должна работать по уже сгенерированному HTML и добавлять классы вокруг известных паттернов только для `longform`.
6. `template.html` или новая версия template должна дать page shell и навигационные элементы.
7. `dark-theme.css` или новая версия CSS должна стать longform-theme, но classic-файлы должны остаться доступными.

Минимальный rollback-контракт:

- `classic` собирает HTML старым template/CSS без семантической постобработки.
- `longform` собирает HTML новой темой и постобработкой.
- Выбор темы должен быть явным через runner/assemble option или env-переменную.
- Если `longform` ломается, оператор может собрать тот же Markdown через `classic` без ручного восстановления файлов.

Почему не чистый CSS:

- Pandoc HTML не знает, где кейс, где план действий, где глава конспекта;
- без классов дизайн быстро скатится в “покрасили заголовки”.

Почему не полноценный app:

- `/notes` должен выдавать переносимый offline HTML;
- интерактивные режимы увеличат риск поломок и тестовую поверхность;
- текущая задача — улучшить чтение, а не строить отдельный продукт.

## Проверка качества

Минимальная проверка после реализации:

- HTML собирается на `test-fixtures/multi-chunk`;
- HTML собирается в `classic` и `longform` режимах;
- classic-режим сохраняет старую структуру без longform-классов;
- longform-режим добавляет семантические классы;
- кликабельные YouTube timestamp links продолжают работать;
- страница открывается локально без сервера;
- mobile ширина не ломает содержание и таблицы;
- print mode остается читаемым;
- существующие runtime/assemble тесты проходят;
- визуальная smoke-проверка в браузере подтверждает: читать стало легче, не хуже.

## Риски

- Слишком агрессивная постобработка может неверно классифицировать обычный текст как кейс или заметку.
- Слишком много визуальных callout-ов снова превратит страницу в карточки.
- Appendix может стать слишком тяжелым визуально, если таблицы и action plan не отделить от основного чтения.

Ограничение против этих рисков: основной текст всегда важнее декоративного слоя. Любое выделение, которое ухудшает последовательное чтение, нужно убрать.

## Решение для реализации

Реализацию стоит делать инкрементально:

1. Сначала page shell, typography, chapter rhythm и appendix styling.
2. Потом семантические классы для case/quote/time/action.
3. Потом browser QA и корректировка плотности.

Не начинать с интерактива.
